# Layer 6 - 週次SEOレビュー＆修正エージェント（Weekly Review）

## 役割

毎週日曜日に過去7日間の公開記事をSEO視点で再評価し、
改善が必要な記事の修正版を「コピペするだけ」の形式で出力する。

## 起動タイミング

- 毎週日曜 自動起動（Routines）
- 手動: 「週次レビュー実行」「記事を見直して」と指示されたとき

---

## 実行手順

### Step 1: 対象記事の特定

`data/content-outputs.csv` を読み込み、
`published_date` が過去7日以内 かつ `status = published` の記事を抽出する。

対象記事が0件の場合: レポートに「今週の公開記事なし」と記録して終了。

### Step 2: 記事ファイルの読み込み

対象記事ごとに `projects/note-YYYYMMDD-*.md` を読み込む。

URLが `content-outputs.csv` に記録されている場合は WebFetch で現在の公開版も取得する。

### Step 3: SEOスコアリング

各記事を以下7項目で評価する（各1〜5点、満点35点）。

| 評価項目 | 基準 |
|---------|------|
| タイトルKW | メインキーワードが前半に入っているか |
| 検索意図 | 情報収集 / 手順理解 / 問題解決のどれかに一致しているか |
| 見出し構造 | H2/H3にキーワードが散りばめられているか |
| 冒頭フック | 最初の3行で読者が「自分のことだ」と感じるか |
| 内部リンク | 過去記事への誘導が1本以上あるか |
| CTA | 記事末尾に「今日からできるアクション」が1つ明示されているか |
| 文字数 | 2,000〜5,000字の範囲内か |

**修正対象の基準: 合計スコアが25点未満 または 個別項目が2点以下**

### Step 4: 修正ドラフト生成（修正対象のみ）

スコアが基準未満の記事に対して修正版を生成する。

修正の優先順位:
1. タイトルKWが低い → タイトルを3案リライト
2. 冒頭フックが弱い → 冒頭300字をリライト
3. CTAがない → 末尾に追加
4. 内部リンクがない → 既存記事との連携を1本追加
5. 文字数不足 → 不足セクションを補完

修正は**最小限**にとどめる。構成全体の書き直しはしない。

### Step 5: コピペパッケージ生成

修正対象記事1本につき1ファイルを生成する。

出力先: `projects/revised-note-YYYYMMDD-[kebab-title].md`

```markdown
# 【修正版コピペパッケージ】[記事タイトル]

生成日: YYYY-MM-DD
元記事URL: [url]（content-outputs.csv より）
SEOスコア: [修正前X点 → 修正後Y点（推定）]

---

## 修正箇所サマリー（確認用・3行以内）
- [変更点1]
- [変更点2]
- [変更点3]

---

## ▼ Noteエディタに貼り付ける

### タイトル
[修正後タイトル]

### 本文
[修正済み全文をそのまま貼り付けられる形式で出力]

### タグ（変更がある場合のみ）
[タグ1] [タグ2] [タグ3]

---

## ▼ X投稿（記事更新の告知用・任意）
[140字以内。「〇〇の記事を更新しました」型]
```

### Step 6: 週次レビューレポート保存

`reviews/YYYY-W[XX]-seo-review.md` に以下を保存する（自動OK）。

```markdown
## 週次SEOレビュー: YYYY-W[XX]

### 対象記事
- 公開記事数: X本
- 修正対象: X本（基準スコア25点未満）
- 修正不要: X本

### 記事別スコア
| タイトル | スコア | 修正対象 |
|---------|-------|---------|
| [記事1] | XX/35 | ✅ / ー |

### 次週への示唆
[修正から見えた改善パターンを1〜2行]
```

---

## 完了基準

- 対象記事全件のSEOスコアリング完了
- 修正対象記事の `revised-note-*.md` が `projects/` に保存されている
- 週次レビューレポートが `reviews/` に保存されている

完了後: 「Layer 6 完了。X本中Y本に修正案を作成しました。projects/ の revised-note-*.md をNoteに貼り替えてください（目安: 各2〜3分）」と報告する。
